Aegis Documentation

Aegis is a vendor-agnostic secrets broker. Applications fetch credentials through a single authenticated API rather than connecting directly to CyberArk, HashiCorp Vault, AWS Secrets Manager, or Conjur. Access is governed by policy, every request is logged, and credentials are never cached.

ℹ

Aegis stores no plaintext credentials. It stores object definitions (which secret to fetch and from where) and API keys (hashed with SHA-256). Plaintext keys are shown once at issuance and never again.

Quick Start

1. Create an Object

An object defines a secret: where it lives and how to fetch it.

curl -X POST /admin/api/objects \
  -H "Authorization: Bearer <admin-token>" \
  -H "Content-Type: application/json" \
  -d '{
    "name":     "db_password",
    "vendor":   "vault",
    "auth_ref": "prod-vault",
    "path":     "secret/data/myapp/db"
  }'

2. Create a Registry

A registry is a named collection of objects that can be granted to teams.

curl -X POST /admin/api/registries \
  -H "Authorization: Bearer <admin-token>" \
  -d '{"name": "production-db"}'

3. Add Objects to the Registry

curl -X POST /admin/api/registries/<reg-id>/objects \
  -H "Authorization: Bearer <admin-token>" \
  -d '{"object_name": "db_password"}'

4. Create a Team and Assign the Registry

curl -X POST /admin/api/teams -d '{"name": "backend"}' ...
curl -X POST /admin/api/teams/<team-id>/registries/<reg-id> ...

5. Fetch Secrets

Use the API key shown at assignment time. Include a change/ticket number for auditability.

curl https://aegis.example.com/secrets \
  -H "Authorization: Bearer sk_..." \
  -H "X-Change-Number: CHG-0042"

Response:

{
  "db_password": "hunter2",
  "_meta": {
    "registry": "production-db",
    "fetched_at": "2026-03-15T09:00:00Z"
  }
}

Authentication

Developer API key

Applications authenticate with a per-team-registry API key. Keys are prefixed sk_ and must be sent as a Bearer token:

Authorization: Bearer sk_<token>

Keys are generated when a team is assigned to a registry. A new key is issued on every rotation. The old key is immediately revoked.

Admin session token

The Admin Panel and Admin API use session tokens obtained by POST /api/login. Sessions expire after 8 hours (configurable).

POST /api/login
{"username": "admin", "password": "..."}
→ {"token": "...", "role": "admin", ...}

Pass the token as a Bearer header on all /admin/api/* calls.

HTTP Basic (CI/CD)

Admin API endpoints also accept HTTP Basic auth with username admin and the ADMIN_PASSWORD environment variable.

Registries

A registry is a named collection of objects. Teams are granted access to entire registries (not individual objects). This allows you to group related secrets and rotate access at the registry level.

✓

Group secrets by environment and service, e.g. prod-db, staging-db, prod-cache. Each team gets a separate API key per registry.

Teams & API Keys

A team represents a consumer of secrets — a service, CI pipeline, or application. Teams are assigned to registries; each assignment produces a unique API key.

User membership

Operator accounts (role: user) belong to one or more teams via the user_teams junction table. Team members can log in to the team dashboard to manage their own webhooks, notification channels, and view their audit activity. Admins manage team membership via POST /admin/api/teams/{id}/members/{user_id}.

Team ID

Every team has a stable UUID. Team members can see their team ID on the dashboard (with a copy button). This ID is used as the path for the inbound webhook URL (/api/inbound/{team_id}) and as the ?team_id= parameter for self-service API calls.

Key lifecycle

When a team is assigned to a registry, Aegis generates a sk_-prefixed key, stores only its SHA-256 hash, and shows the plaintext exactly once. There is no way to retrieve it again — rotate instead.

Key suspension

Admins can temporarily suspend a key without revoking it. A suspended key is rejected at /secrets with 401 (indistinguishable from a revoked key to the caller) but can be re-enabled without issuing a new key. Use this for incident response when you need to block a team immediately without burning their key.

Multiple registries

A team can be assigned to multiple registries. Each assignment has its own key, so a compromise of one key does not expose other secrets.

Policy Engine

Policies constrain GET /secrets access. They can be set at the registry level (applies to all teams accessing that registry) or the team level (applies to that team for all its registries). Team policies are checked first; then registry policies.

Policy fieldDescription
ip_allowlistCIDR blocks the request source IP must match. Empty = unrestricted.
allowed_from / allowed_toUTC time window. Requests outside the window are blocked.
cn_requiredIf true, requests must include a client certificate with a matching CN.
rate_limit_rpmMaximum requests per minute from this team. Enforced via Redis.
max_key_daysRegistry-only. API keys expire after this many days and are auto-rotated.
⚠

Policy violations fire a policy.violated webhook event if a webhook is configured for the team.

Key Lifecycle

Active

↔
Active
→
Expiring Soon
→

Issued: Key is generated when a team is assigned to a registry. Plaintext shown once.

Active: Key is valid. Every GET /secrets validates the hash.

Suspended: Admin has temporarily disabled the key via PATCH /admin/api/keys/{key_id}/suspend. Requests are rejected with 401 (same as revoked — no information leak). The key can be re-enabled without rotation. Useful for incident response.

Expiring soon: Within key_warning_days of expires_at, Aegis fires key.expiring_soon webhook. At most once per 23 hours.

Auto-rotated: When expires_at is reached (checked daily by the scheduler), the old key is revoked and a new one is issued. A key.rotated webhook is fired containing the new plaintext key — your endpoint must persist it.

Revoked: Key is permanently invalid. Revocation happens on manual rotation or when a team is unassigned from a registry.

⚠

For auto-rotation to deliver the new key, ensure the key.rotated webhook event is subscribed and the endpoint is reachable. If delivery fails, the key is still rotated — you will need to rotate manually from the admin panel to get a new plaintext.

Webhooks & Notifications

Each team can have one outgoing webhook endpoint. Aegis will POST JSON to that URL when subscribed events occur. Delivery is attempted up to 3 times (immediately, then after 5s, then 30s). Teams can also receive human-readable alerts via Slack, MS Teams, or Discord notification channels.

Outgoing webhooks (machine-to-machine)

Outgoing webhooks deliver structured JSON events to an HTTP endpoint of your choice. Configure them in the admin panel (admin-only) or via the team self-service dashboard (PUT /api/my-webhook).

Notification channels (human alerts)

Teams can configure up to three notification channels for human-readable alerts on key events:

HMAC signing (optional)

When signing is enabled for a webhook, Aegis adds an X-Aegis-Signature header:

X-Aegis-Signature: sha256=<hex-digest>

Verify it in your endpoint:

import hmac, hashlib

def verify(payload_bytes: bytes, secret: str, signature_header: str) -> bool:
    expected = "sha256=" + hmac.new(
        secret.encode(), payload_bytes, hashlib.sha256
    ).hexdigest()
    return hmac.compare_digest(expected, signature_header)
ℹ

Signing is opt-in. Without it, Aegis still POSTs the event with X-Aegis-Event header — suitable for internal endpoints on private networks. Enable signing for public-facing endpoints.

Inbound webhooks (CI/CD → Aegis)

Aegis also accepts inbound webhook calls from CI/CD pipelines. Each team has a stable inbound URL based on their team ID:

POST /api/inbound/{team_id}

Requests must be HMAC-SHA256 signed using the team's webhook signing secret (same secret as the outgoing webhook). Supported actions:

actionDescriptionResponse
pingConnectivity check{"ok": true, "action": "ping"}
rotate_keyTrigger key rotation for a specific registryNew plaintext key + preview
 
# Ping from a CI pipeline
curl -X POST https://aegis.example.com/api/inbound/<team-id> \
  -H "Content-Type: application/json" \
  -H "X-Aegis-Signature: sha256=<hmac>" \
  -d '{"action": "ping"}'

# Rotate key for a registry
curl -X POST https://aegis.example.com/api/inbound/<team-id> \
  -H "Content-Type: application/json" \
  -H "X-Aegis-Signature: sha256=<hmac>" \
  -d '{"action": "rotate_key", "registry_id": "<reg-id>"}'
⚠

The inbound webhook requires an active webhook with signing_enabled: true. Requests with missing or invalid signatures return 401.

SIEM & Audit

Every GET /secrets call is written to the audit log. Fields are snapshotted at request time so audit records remain accurate even after entities are renamed or deleted.

Audit events are emitted to configured log destinations (set via LOG_DESTINATIONS env var):

DestinationDescription
stdoutJSON lines to container stdout (default)
s3Batched to S3 bucket (configure via LOG_S3_BUCKET, LOG_S3_PREFIX)
siemForward to a SIEM HTTP endpoint (LOG_SIEM_URL)

The change log records every admin mutation (create/update/delete of objects, registries, teams, users, settings). Both logs are immutable — there is no delete endpoint.

Admin API — Key Suspension

Suspend or re-enable a team's API key without revoking it. A suspended key returns 401 at /secrets, indistinguishable from a revoked key. It can be re-enabled at any time without rotation.

PATCH /admin/api/keys/{key_id}/suspend Toggle key suspension (enable ↔ disable) ▶

Toggles suspended on the key. No body required.

{
  "key_id":      "uuid",
  "suspended":   true,
  "key_preview": "sk_abc123..."
}

The mutation is recorded in the change log. The team's webhook fires no event for suspension — this is intentional (avoid alerting an attacker that their key was blocked).
